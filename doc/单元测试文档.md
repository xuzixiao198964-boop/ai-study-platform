# 学习指认AI — 单元测试文档

**产品名称**：学习指认AI（K12 AI 辅助学习平台）  
**文档版本**：1.1  
**适用范围**：后端（FastAPI）与前端（Flutter）单元级验证  

---

## 1. 文档目的

本说明定义「学习指认AI」平台单元测试的范围、策略、用例清单与执行方式，用于在隔离环境下验证各模块逻辑正确性，为持续集成与发布门禁提供依据。

---

## 2. 测试策略

### 2.1 技术选型

| 层级 | 工具 | 说明 |
|------|------|------|
| 后端 | **pytest** + **pytest-asyncio** | 覆盖异步路由、服务层与工具函数；配合 `httpx.AsyncClient` 或 `TestClient` 调用 FastAPI 应用 |
| 前端 | **Flutter test** | `test/` 目录下 Widget 测试与纯 Dart 单元测试；关键 Provider、Service 与视图逻辑优先 |
| 覆盖率 | **pytest-cov** / Flutter coverage | 以行覆盖为主，关注分支与错误路径 |

### 2.2 设计原则

- **隔离外部依赖**：数据库、Redis、第三方 AI（DeepSeek 等）通过 fixture 替换为内存实现、Fake 或 Mock，保证测试稳定、可重复。
- **单一职责**：每个用例验证一个明确行为；失败时能快速定位模块与用例编号。
- **命名规范**：测试文件与函数名与业务模块对应；用例 ID（如 `TC-AUTH-001`）在文档与代码注释或 `@pytest.mark` 中可追溯。

### 2.3 质量目标

- **目标代码覆盖率**：全项目 **≥ 80%**（后端以 `app` 包为主；前端以 `lib/` 核心业务路径为主）。
- **门禁建议**：合并请求前在 CI 中执行单元测试；覆盖率低于阈值时告警或阻断（按团队规范配置）。

### 2.4 测试分层说明

| 层级 | 内容 | 非本文件范围 |
|------|------|----------------|
| 单元测试 | 函数/类/单接口逻辑 | 多服务联调、真实部署验证见《集成测试文档》 |

## 3. 后端测试用例清单

以下用例均在 **Mock 或测试数据库** 环境下执行；认证类用例需构造合法 JWT 或依赖 `get_current_user` 的测试覆盖。

### 3.1 认证模块（auth）

| 用例 ID | 用例名称 | 前置条件 / 输入要点 | 预期结果 |
|---------|----------|---------------------|----------|
| **TC-AUTH-001** | 用户注册 — 正常注册成功 | 唯一用户名、符合策略的密码 | HTTP 200，返回访问令牌或约定结构；用户落库 |
| **TC-AUTH-002** | 用户注册 — 用户名已存在 | 与已有用户同名 | HTTP 4xx，`detail` 表明冲突 |
| **TC-AUTH-003** | 用户注册 — 密码太短 | 密码长度低于 `MIN_PASSWORD_LENGTH` | HTTP 422 或 400，校验错误信息明确 |
| **TC-AUTH-004** | 用户登录 — 正常登录 | 已注册用户名与正确密码 | HTTP 200，返回有效 `access_token` |
| **TC-AUTH-005** | 用户登录 — 密码错误 | 正确用户名、错误密码 | HTTP 401，无有效令牌 |
| **TC-AUTH-006** | 获取当前用户 — token 有效 | 请求头携带未过期 Bearer Token | HTTP 200，返回当前用户信息与角色 |
| **TC-AUTH-007** | 获取当前用户 — token 过期 | 过期或伪造 JWT | HTTP 401 |
| **TC-AUTH-008** | 修改密码 | 已登录 + 旧密码正确 + 新密码符合策略 | 修改成功；旧密码不可再登录 |

**关联接口（参考）**：`POST /api/v1/auth/register`、`POST /api/v1/auth/login`、`GET /api/v1/auth/me`、`POST /api/v1/auth/change-password` 等。

### 3.2 AI 对话模块（chat）

| 用例 ID | 用例名称 | 前置条件 / 输入要点 | 预期结果 |
|---------|----------|---------------------|----------|
| **TC-CHAT-001** | 发送普通对话消息 | 已登录、有效 `session_id`、普通文本 | 返回助手回复；消息持久化（若设计如此） |
| **TC-CHAT-002** | 场景切换检测 — 「做作业」→ camera 模式 | 用户消息含触发关键词（如「我要做作业」） | 响应或元数据中 `mode`（或等价字段）为 camera |
| **TC-CHAT-003** | 场景切换检测 — 「关掉摄像头」→ chat 模式 | 用户消息含关闭摄像头类表述 | 模式切回 chat |
| **TC-CHAT-004** | 获取对话历史 | 已有会话与多条消息 | `GET history` 返回按序列表，分页/条数符合约定 |
| **TC-CHAT-005** | 对话上下文保持 | 连续多条消息引用前文 | 后续回复与上下文一致（可对 AI 层 Mock 固定输出断言） |

**关联接口（参考）**：`POST /api/v1/chat/message`、`GET /api/v1/chat/history/{session_id}`。

## 6. 执行方式

```bash
# 后端（于 server 目录；默认不跑 integration）
cd server && pytest tests/ -v --cov=app --cov-report=html

# 仅运行集成标记（需配置环境变量，见《集成测试文档》）
cd server && pytest tests/ -v -m integration

# 前端（于 mobile 目录）
cd mobile && flutter test
```

**可选**：生成 Flutter 覆盖率报告可使用 `flutter test --coverage`，再用 `lcov` 工具生成 HTML（按团队 CI 配置）。
